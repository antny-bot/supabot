# 사용자 온보딩 가이드

이 문서는 `supabot`에 새로운 사용자를 등록하고, 필요하면 `supabot-manager`까지 사용할 수 있게 여는 절차를 정리한 운영 가이드다.

## 한눈에 보는 흐름

1. 사용자가 텔레그램에서 봇에게 `/start`를 보낸다.
2. 봇이 사용자를 `pending` 상태로 등록하고, 관리자에게 승인 필요 알림을 보낸다.
3. 관리자가 `supabot-manager`의 사용자 화면에서 해당 사용자를 `active`로 승인한다.
4. 사용자가 다시 텔레그램 봇을 열고 기능을 사용한다.
5. 웹 대시보드도 써야 하면, 관리자가 `manager_email`을 연결하고 Supabase Auth 계정을 만든다.
6. 사용자가 `supabot-manager`에 로그인한다.

## 1. 사전 준비

다음이 먼저 준비되어 있어야 한다.

- `supabot` 봇이 실행 중이어야 한다.
- `supabot-manager`를 쓸 경우 manager도 배포되어 있어야 한다.
- 봇과 manager는 같은 Supabase 프로젝트를 바라봐야 한다.
- 관리자는 봇 또는 manager에서 사용자 승인 권한을 가져야 한다.

배포 자체는 아래 문서를 참고한다.

- 봇/전체 구조: [README.md](/E:/apps/supabot/README.md)
- manager 배포: [manager/README.md](/E:/apps/supabot/manager/README.md)

## 2. 텔레그램 봇 사용자 등록

### 사용자가 할 일

1. 텔레그램에서 봇을 연다.
2. `/start`를 보낸다.
3. 봇이 "등록 요청이 접수되었고 관리자 승인 후 사용할 수 있다"는 안내를 보여주면 대기한다.

현재 구현상 `/start`를 처음 보내면 사용자는 자동 생성되지만 상태는 `pending`이다. 아직 승인 전이면 다른 기능을 바로 쓸 수 없다.

### 관리자가 할 일

1. `supabot-manager`의 사용자 목록으로 간다.
2. 새로 들어온 `pending` 사용자를 찾는다.
3. 확인 후 `Approve` 또는 활성화 동작으로 상태를 `active`로 바꾼다.

승인되면 manager는 해당 사용자에게 텔레그램 알림을 보낸다. 이후 사용자는 텔레그램에서 봇 기능을 정상적으로 사용할 수 있다.

## 3. manager까지 열어줘야 하는 경우

텔레그램만 쓰는 사용자는 여기까지면 충분하다. 웹 대시보드 `supabot-manager`도 써야 하면 아래 절차가 추가로 필요하다.

### 3-1. 관리자: 로그인 이메일 연결

1. `supabot-manager` 사용자 화면에서 대상 사용자를 연다.
2. 그 사용자의 `manager_email`에 로그인에 쓸 이메일을 저장한다.

이 이메일은 봇 사용자와 manager 로그인을 연결하는 키다. manager는 로그인 후 `users.manager_email`과 일치하는 사용자를 찾아 세션을 연다.

### 3-2. 관리자: Supabase Auth 계정 생성

**권장: manager 화면에서 초대 메일 발송**

1. `supabot-manager` 사용자 화면에서 대상 사용자를 연다.
2. `manager_email`이 저장되어 있으면 `초대 메일 발송` 버튼이 보인다.
3. 버튼을 누르면 Supabase가 해당 이메일로 Auth 계정을 생성하고, 사용자가 직접 비밀번호를 설정할 수 있는 초대 링크를 메일로 발송한다.
4. 사용자가 메일의 링크로 접속해 비밀번호를 설정하면 계정이 활성화된다.
5. 초대 메일을 한 번 발송하면(=`manager_invited_at` 기록됨) 같은 사용자 행의 버튼은 `초대 메일 발송`에서 `비밀번호 재설정`으로 바뀐다. 사용자가 비밀번호를 잊었을 때 이 버튼으로 재설정 메일을 다시 보낼 수 있다 (중복 초대 방지).

> 이 기능은 Supabase 프로젝트에 `Authentication > Email Templates` / SMTP가 설정되어 있어야 정상적으로 메일이 발송된다. 메일이 도착하지 않으면 프로젝트의 메일 발송 설정을 먼저 확인한다.

**대안: Supabase 대시보드에서 수동 생성**

위 방법을 쓸 수 없는 경우(SMTP 미설정 등) 아래처럼 수동으로 만들 수도 있다.

1. Supabase 대시보드로 간다.
2. `Authentication > Users`에서 새 사용자를 만든다.
3. 위에서 연결한 것과 같은 이메일 주소를 사용한다.
4. 초기 비밀번호를 설정한다.
5. 필요하면 `Auto Confirm`를 켠다.

중요한 점:

- `manager_email`에 넣은 이메일과 Supabase Auth 계정 이메일이 같아야 한다.
- 둘이 다르면 manager 로그인은 성공해도 해당 봇 사용자와 연결되지 않는다.

### 3-3. 사용자: manager 로그인

1. 관리자에게 받은 이메일/비밀번호로 `supabot-manager` 로그인 화면에 접속한다.
2. 처음 로그인에 성공하면 연결된 봇 사용자 권한으로 대시보드를 사용한다.

현재 구현 기준으로 MFA가 켜져 있으면 로그인 뒤 OTP 2차 인증이 추가로 필요하다.

## 4. 권장 운영 순서

새 사용자 1명을 열 때는 아래 순서를 권장한다.

1. 사용자가 텔레그램 `/start`
2. 관리자가 사용자를 확인하고 `active` 승인
3. 텔레그램 기능 사용 확인
4. manager가 필요하면 `manager_email` 설정
5. Supabase Auth 계정 생성
6. manager 로그인 확인
7. 필요하면 MFA 설정

이 순서가 좋은 이유는, 텔레그램 사용자 식별자(`user_id`)가 먼저 생겨야 manager 계정을 어떤 봇 사용자에 연결할지 명확해지기 때문이다.

## 5. 자주 막히는 지점

### 텔레그램에서 `/start` 했는데 계속 대기 상태인 경우

- 사용자가 아직 `pending`일 가능성이 높다.
- manager 사용자 화면에서 상태가 `active`인지 확인한다.

### manager 로그인이 안 되는 경우

아래를 순서대로 확인한다.

1. Supabase Auth 계정이 실제로 생성되었는지
2. 로그인 이메일과 `manager_email` 값이 정확히 같은지
3. 비밀번호가 맞는지
4. manager가 올바른 Supabase 프로젝트를 보고 있는지

### manager는 로그인되는데 원하는 사용자 데이터가 안 보이는 경우

- 대개 `manager_email`이 잘못 연결된 경우다.
- 다른 사용자에 같은 이메일이 이미 연결되었는지도 확인한다.

### MFA 때문에 로그인 진행이 안 되는 경우

- 현재 구현에서는 MFA 설정/비활성화 UI가 대시보드 안에 있다.
- OTP 앱 코드가 필요한 상태인지 확인한다.
- 필요하면 관리자가 해당 사용자의 MFA 운영 정책을 별도로 안내해야 한다.

## 6. 운영 체크리스트

- 텔레그램 `/start`로 사용자가 먼저 생성되었는가
- 사용자 상태가 `active`인가
- 거래소 키 등록 전 승인 정책이 정해져 있는가
- manager가 필요하면 `manager_email`이 설정되었는가
- 같은 이메일로 Supabase Auth 계정이 생성되었는가
- 로그인과 MFA 동작을 실제로 한 번 확인했는가
